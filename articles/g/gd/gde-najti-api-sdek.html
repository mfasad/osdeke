Где найти API СДЭК: адреса серверов и документация

Поиск официальной документации и рабочих адресов для API СДЭК — это первостепенная задача для разработчиков, интегрирующих логистику в интернет-магазин или CRM-систему. В отличие от простых сервисов, где достаточно найти кнопку «получить ключ», корпоративная интеграция требует понимания архитектуры взаимодействия между серверами. Часто пользователи теряются в обилии версий протоколов, путая устаревшие методы с актуальными REST-решениями, что приводит к ошибкам при тестировании.

Основным и единственным авторитетным источником информации является портал разработчиков компании. Именно там сосредоточены все необходимые спецификации, Swagger-файлы и инструкции по авторизации. Важно понимать, что доступ к продуктивной среде (production) возможен только после заключения договора, однако для отладки кода предоставляются специальные тестовые шлюзы.

В этой статье мы подробно разберем, где именно искать актуальные ссылки, как получить необходимые учетные данные и чем отличаются различные версии API. Вы узнаете, как правильно настроить окружение для разработки, чтобы избежать блокировок и ошибок соединения на ранних этапах внедрения.

Официальный портал разработчика CDEK Dev

Центральным хабом для всех технических специалистов, работающих с логистическими сервисами, является сайт cdek.dev. Это специализированная платформа, созданная специально для разработчиков, где публикуется вся актуальная документация. Именно здесь, а не на общем сайте транспортной компании, находятся исчерпывающие сведения о методах запросов, форматах JSON-ответов и кодах ошибок.

На портале представлена детальная информация по всем основным направлениям интеграции. Вы сможете найти руководства по расчету стоимости доставки, оформлению заказов, отслеживанию статусов и работе с картой пунктов выдачи. Структура сайта построена логично, позволяя быстро перейти к нужному разделу, будь то API 2.0 или специфические модули для маркетплейсов.

⚠️ Внимание: Использование документации со сторонних ресурсов или форумов может привести к неработоспособности вашего модуля, так как транспортная компания регулярно обновляет протоколы безопасности и структуру данных.

Для начала работы вам потребуется зарегистрировать личный кабинет разработчика. Это действие бесплатно и открывает доступ к тестовым серверам, где можно эмулировать реальные операции без финансовых затрат. После регистрации в личном кабинете генерируются уникальные идентификаторы, необходимые для авторизации каждого запроса.

💡

Только портал cdek.dev содержит актуальную и полную документацию по всем версиям API, включая методы авторизации OAuth 2.0.

Различия между API 1.0 и API 2.0

При поиске технических спецификаций критически важно понимать, с какой версией протокола вы работаете. Компания активно мигрирует пользователей на API 2.0, которое базируется на архитектуре REST и стандартах безопасности OAuth 2.0. Старая версия (1.0), часто называемая XML-API, постепенно выводится из эксплуатации и имеет ограниченный функционал.

Новая версия предлагает более гибкую работу с данными, поддержку JWT-токенов и расширенный набор методов для управления заказами. В то время как legacy-версия требовала передачи параметров в URL или XML-теле запроса, современный стандарт предполагает работу с JSON-объектами, что упрощает парсинг ответов в большинстве языков программирования.

Основные отличия заключаются не только в формате данных, но и в механизме авторизации. Если раньше использовалась простая пара логин-пароль в каждом запросе, то теперь требуется процедура получения временного токена доступа. Это повышает безопасность передаваемой информации, но требует реализации дополнительного шага в коде вашего приложения для обновления токена.

  • 🔑 Авторизация: В версии 2.0 используется OAuth 2.0 с получением JWT-токена, тогда как 1.0 использовала базовую авторизацию.
  • 📦 Формат данных: Современный стандарт работает исключительно с JSON, уходя от устаревшего XML.
  • 🚀 Производительность: REST API 2.0 оптимизировано для высоких нагрузок и быстрее обрабатывает запросы.

Если вы только начинаете интеграцию, нет смысла тратить время на изучение устаревших методов. Все новые функции и улучшения внедряются исключительно в API 2.0. Поддержка первой версии будет прекращена, поэтому инвестиции времени в изучение актуального стека технологий являются наиболее рациональным решением для вашего бизнеса.

Адреса серверов: Production и Sandbox

Один из самых частых вопросов при настройке окружения — куда именно отправлять HTTP-запросы. Адреса серверов строго разделены на две среды: тестовую (Sandbox) и боевую (Production). Использование адресов продакшена для тестирования недопустимо, так как это может привести к созданию реальных заказов и финансовым потерям.

Для среды Sandbox предназначен отдельный шлюз, который эмулирует работу реальной системы, но не проводит финансовые транзакции и не создает реальных накладных. Здесь можно бесконечно тестировать сценарии оформления, отмены и изменения заказов, не опасаясь последствий. Базы данных тестовой среды периодически очищаются, поэтому не стоит использовать её для хранения важных данных.

Боевая среда (Production) доступна только после заключения договора и получения соответствующих учетных данных. Адреса серверов продакшена отличаются от тестовых, и ошибка в одной букве доменного имени приведет к ошибке соединения. Ниже приведена таблица с актуальными адресами для различных методов взаимодействия.

Среда Назначение Базовый URL (API 2.0)
Sandbox Тестирование и отладка https://api.edu.cdek.ru/v2
Production Работа с реальными заказами https://api.cdek.ru/v2
Auth (OAuth) Получение токенов https://api.cdek.ru/oauth

При разработке модуля необходимо предусмотреть возможность переключения между этими адресами через конфигурационный файл. Это позволит легко переносить приложение из тестового режима в рабочий без изменения исходного кода. Убедитесь, что ваш фаервол или сервер безопасности разрешает исходящие соединения по порту 443 (HTTPS) на указанные домены.

Всегда проверяйте заголовок ответа сервера (Header), чтобы убедиться, что вы находитесь в нужной среде (test или prod), особенно при работе с несколькими проектами одновременно.

Получение учетных данных и авторизация

Для доступа к API вам потребуется набор уникальных идентификаторов. Процесс их получения зависит от стадии разработки. На этапе тестирования вы можете самостоятельно зарегистрироваться на портале разработчика и мгновенно получить доступ к песочнице. Для работы в боевой среде потребуется договор с транспортной компанией.

Ключевыми элементами безопасности являются Account ID (идентификатор аккаунта), Secure Key (секретный ключ) и, в некоторых случаях, Secure Password. Эти данные передаются при запросе токена доступа. Никогда не публикуйте эти ключи в открытых репозиториях, таких как GitHub, и не передавайте их третьим лицам без необходимости.

Процесс авторизации в API 2.0 выглядит следующим образом: ваше приложение отправляет POST-запрос с учетными данными на сервер авторизации. В ответ сервер возвращает access_token и время его жизни. Этот токен необходимо использовать в заголовке Authorization для всех последующих запросов к бизнес-логике.

⚠️ Внимание: Токен доступа имеет ограниченное время жизни. Ваше приложение должно автоматически запрашивать новый токен до истечения срока действия старого, чтобы избежать прерывания работы сервиса.

Храните секретные ключи в переменных окружения вашего сервера, а не в коде программы. Это стандартная практика безопасности, которая защищает данные даже в случае утечки исходного кода. Для локальной разработки используйте файлы .env, добавив их в .gitignore.

Где скачать Swagger и примеры кода

Для ускорения процесса интеграции разработчики могут воспользоваться интерактивной документацией Swagger (OpenAPI). Она позволяет не только читать описание методов, но и тестировать их прямо в браузере, отправляя тестовые запросы и получая ответы. Это значительно упрощает понимание структуры требуемых данных.

Файлы спецификации swagger.json или yaml можно скачать напрямую с портала разработчика. Они совместимы с большинством IDE и инструментов для работы с API, таких как Postman или Insomnia. Импортировав спецификацию в Postman, вы получите готовую коллекцию запросов, которую останется только наполнить своими токенами.

Также на сайте доступны примеры кода на популярных языках программирования: PHP, Python, Java, C#. Эти сниппеты демонстрируют базовые сценарии использования, такие как расчет стоимости или создание заказа. Однако стоит помнить, что это лишь примеры, и в реальном проекте их необходимо адаптировать под архитектуру вашего приложения.

  • 📄 Swagger UI: Визуальный интерфейс для изучения методов API.
  • 💻 SDK: Готовые библиотеки для упрощения интеграции в популярных фреймворках.
  • 📝 Postman Collection: Готовый набор запросов для быстрого старта тестирования.

Использование готовых библиотек (SDK) может сократить время разработки, но убедитесь, что вы используете последнюю версию. Разработчики сообщества часто обновляют пакеты, добавляя поддержку новых полей или исправляя баги, связанные с изменениями на стороне сервера логистической компании.

Что делать, если Swagger не открывается?

Если документация Swagger не загружается, проверьте блокировщики рекламы или корпоративный фаервол. Также попробуйте использовать режим инкогнито или очистить кэш браузера, так как скрипты документации могут кэшироваться некорректно.

Типичные ошибки при интеграции

Даже имея доступ к отличной документации, разработчики часто сталкиваются с типовыми проблемами. Одной из самых распространенных ошибок является неверный формат передаваемых данных. Например, даты должны быть в строго определенном формате YYYY-MM-DD, а телефонные номера — в международном формате без лишних символов.

Другая частая проблема — игнорирование лимитов частоты запросов (Rate Limiting). Серверы транспортной компании ограничивают количество запросов в секунду для защиты от перегрузок. Если ваше приложение превышает лимит, вы получите ошибку 429 Too Many Requests. Необходимо реализовать механизм повторных попыток (retry logic) с экспоненциальной задержкой.

Также стоит внимательно относиться к кодам городов и тарифов. Они могут меняться или выводиться из употребления. Всегда получайте актуальные справочники через API перед формированием заказа, а не храните их в статическом виде в базе данных годами.

⚠️ Внимание: Логируйте все запросы и ответы API (особенно коды ошибок), это критически важно для отладки. Без логов найти причину отказа в создании заказа будет практически невозможно.

Не забывайте проверять статусы заказов через вебхуки (уведомления), если такая возможность предусмотрена, вместо постоянного опроса сервера (polling). Это снизит нагрузку на вашу инфраструктуру и обеспечит более оперативное получение информации об изменении статуса груза.

💡

Успешная интеграция зависит не только от правильного кода, но и от грамотной обработки ошибок и соблюдения ограничений сервера.

Где найти список всех кодов ошибок API?

Полный перечень кодов ошибок с их расшифровкой и возможными решениями находится в разделе"Ошибки" на портале cdek.dev. Там описаны коды от 400 (Bad Request) до специфических бизнес-ошибк, таких как неверный индекс или недоступность тарифа.

Можно ли использовать API СДЭК бесплатно?

Использование тестовой среды (Sandbox) полностью бесплатно и не требует договора. Для работы в боевой среде (Production) необходим заключенный договор с транспортной компанией, условия тарификации услуг зависят от вашего договора.

Как долго хранятся данные в тестовой среде?

Данные в Sandbox не гарантированы для длительного хранения. Базы данных тестовой среды могут быть очищены в любой момент без предупреждения для проведения технических работ или обновления версий ПО.
